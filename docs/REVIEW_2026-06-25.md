## 1. 🟠 medium（先に直す価値が高い）

### M-1 メニューの「未提出」警告と入力ボタンの着地月がズレる（バグ）
- **場所:** `app.py:485`（menu）, `app.py:599`（worker）, `templates/menu.html:49`
- **事象:** メニューは**翌月**の未提出を見て「{{next_m}}月分が未提出です」と促すのに、「シフト希望を入力」ボタンは `url_for('worker')` を**年月なし**で呼ぶ。worker 側は `resolve_ym(now.year, now.month)` で**当月**に解決するため、「7月が未提出」と言われて押すと**6月（当月）の画面**が開く。初心者職員が当月を空提出し、肝心の翌月を出し忘れる事故につながる。
- **改善案（最小・約3行）:** menu ルートで `next_y` もテンプレへ渡し、リンクを `url_for('worker', year=next_y, month=next_m)` にする。これで警告月＝着地月が一致。（代替: worker の既定月を翌月に変える案もあるが影響範囲が広いので非推奨。）

### M-2 管理者のパスワード再発行が、被害者の既存セッションを無効化しない
- **場所:** `app.py:235`（load_current_user）, `app.py:759`（admin_password_set）, `app.py:1015`（change_password）
- **事象:** セッションは `username` のみ保持し、各リクエストで `role/name/is_active` だけ再取得する（パスワード世代を持たない）。そのため管理者がパスワードを再設定しても、被害者の既存セッションCookieは**無操作30分／絶対24時間が切れるまで有効**。本人の自己変更も「今の端末」は `session.clear()` するが、**別端末の旧セッションは残る**。アカウント乗っ取り・退職時の即時締め出しができない。
- **改善案:**
  - **(A) 運用で対応（推奨・最小）:** 「ユーザー管理」画面と README に *「パスワード再発行だけでは、ログイン中の端末は最大24時間ログアウトされません。すぐ締め出すには『停止』を使ってください」* と明記。退職・乗っ取り時の手順を **「停止 →（必要なら）パスワード再発行」** に固定。
  - **(B) 恒久対策（任意・約15行）:** `users` に `session_epoch`(整数)列を追加。ログイン時に `session['epoch']` を保存し、パスワード変更／管理者再設定時に epoch をインクリメント、`load_current_user` で不一致なら `session.clear()`。10名規模なら (A) で実害は十分小さい。

## 2. 🔐 セキュリティ（一次ソース検証済み）

### S-1 🟡 `CF-Connecting-IP` を無条件に信頼している（デプロイ前に判断が必要）
- **場所:** `app.py:72`（client_ip）, `settings.py:109`, `render.yaml:37`
- **事実（一次確認済み）:** `client_ip()` は `TRUST_CF_CONNECTING_IP=1` のとき `CF-Connecting-IP` の値を「IPとして妥当か」だけ検証して採用。この値は**ログインのレート制限キー**と**監査ログのIP列**に直結する。Cloudflare 公式は「CF-Connecting-IP はトラフィックが100% Cloudflare 経由だと確信できる場合のみ信頼してよい」と明記。**`CF-Connecting-IP` は Cloudflare 固有のヘッダで、Render のエッジは付与せず**、Render は本来 `X-Forwarded-For` で実IPを渡す。
- **判断の分岐（デプロイ前に確認）:**
  - **自前の Cloudflare を前段に置いていない場合** → ヘッダは攻撃者が偽装可能。`render.yaml` の `TRUST_CF_CONNECTING_IP` を **`0` にする**（`TRUSTED_PROXY_HOPS=1`＋`X-Forwarded-For` で実IPは取れる）。デプロイ後 `/logs` のIPが実クライアントIPらしいか確認。
  - **自前 Cloudflare を前段に置く場合** → Render の受信を **Cloudflare の IP レンジに限定**してエッジ迂回を塞ぐ（そうしないと `*.onrender.com` 直叩きで偽装可能）。
- **補足:** 検証では「Render は全インバウンドを自社 Cloudflare 網経由で受ける」ため実害は緩和されるとし low 評価。ただし**前提が文書化されておらず**、初心者が `=1` のまま別環境へ移すと危険。最低限 `client_ip()` の docstring と CODE_REVIEW.md に「この信頼はエッジ経由でのみ受理する構成が前提」と1〜2行明記を。
- **一次資料:** Cloudflare HTTP headers / Protect your origin server / Render: How Render handles DDoS

### S-2 🟡 ログインのブルートフォース耐性が「IPレート制限」のみ（アカウントロック無し）
- **場所:** `app.py:437`, `security_utils.py:5`（`PASSWORD_MIN_LEN=8`）
- **事実:** 保護は「8文字最小＋弱PW16件ブロック＋IP単位20/分」だけで、**ユーザー単位の連続失敗ロック／指数バックオフは無し**（`login_fail` は監査記録のみで制御に未使用）。レート制限は `memory://`（単一worker前提）でキーは `client_ip()` 依存（S-1と連動）。
- **改善案:**
  - **(必須・文書)** CODE_REVIEW.md の「8文字＝NIST準拠」表記は**不正確**なので訂正。NIST SP 800-63B は単要素パスワードに**アカウント単位のレート制限**（連続失敗≤100で当該認証子を停止）と**15文字 SHALL**（8文字はMFA前提）を求める。「アカウントロックは未実装。保護はIPレート制限＋弱PWブロックのみ」と受容事項として明記。
  - **(任意・軽量)** 既存 `audit_log` の `login_fail` を ID 単位で直近N分集計し、閾値超過時に当該IDへ数秒の応答遅延／一時拒否を入れると、IP分散攻撃や S-1 が崩れた場合の保険になる。
  - 機微情報を扱う運用に拡張するなら 8→15文字化 or MFA を再評価。
- **一次資料:** NIST SP 800-63B

### ⚪ 確認して問題なしだったもの（重要）
一次確認の結果、以下は**いずれも妥当**: パラメータ化SQL（SQLi無し）、Jinja自動エスケープ（remark/name/colorのXSS無し）、`deny_if_not_admin` による管理ルートの認可網羅、ログイン時 `session.clear()→permanent`（固定化対策）、各リクエストでの停止/降格の即時反映、`__Host-` Cookie・無操作/絶対期限、CSP（nonce, `frame-ancestors 'none'`, `form-action 'self'`）、全フォームのCSRF、`MAX_CONTENT_LENGTH`、監査ログへ秘密情報を残さない設計。
