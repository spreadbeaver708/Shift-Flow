## 1. 🟠 medium（実装済み）

### A-1 「/」がログイン済み職員に403エラー
- **場所:** `app.py`（index ルート）、`tests/test_v29.py`
- **事象:** ルートURL「/」は管理者専用（自分の希望入力）で、ログイン済みの職員が開くと **403エラー画面＋authz_fail監査記録**。「/」はブックマーク・ドメイン直打ちで職員が最も踏みやすい入口のため、①初心者がエラー画面で迷子になる ②良性アクセスがauthz_failとして蓄積し、月次のログ確認で本物の異常が埋もれる（警報疲れ）。
- **経緯:** V29の「管理ルート認可統一」で意図的に403化されたもの。管理専用画面の403+authz_failは正しい設計だが、「/」だけは入口URLという性格が異なる。
- **対応（実装済み）:** 未ログイン→ログイン画面、職員→自分の入力画面（/worker）へリダイレクト。管理者は従来どおり。他の管理画面のガード（`deny_if_not_admin`）は不変。テストを新仕様に更新。

### A-2 職員が任意の月（2000〜2100年）へ保存POST可能
- **場所:** `app.py`（worker ルート）、`tests/test_deadline.py`
- **事象:** 画面は今月・翌月しか提供しないが、hidden値を書き換えた直接POSTで職員が**任意の月**に保存できた。締め切りロックは「締め切りが設定された月」しか守らないため、①締め切り未設定の**過去月の書き換え**（管理者が気づきにくい） ②遠い将来月へのごみ登録、が可能だった（監査ログには残る）。
- **対応（実装済み):** 職員のPOSTは**前月・当月・翌月**の3ヶ月窓のみ受理（前月を含めるのは、月替わり深夜0時をまたぐ送信を弾かないため）。窓外は「この月は保存できません。今月か翌月を選んでください。」で入力画面へ。GET閲覧、管理者ルート（`/`・`/staff/<u>`）は無制限のまま（過去月修正の業務ニーズ）。回帰テスト3件追加。
- **副次修正:** `tests/test_deadline.py` が2026年7月をハードコードしており、時間経過で壊れる時限故障だったため、実行時のJST現在月ベースへ書き換えた。

## 3. ⚪ 却下した指摘（クリティカル検証の記録）

並列調査で挙がったが、一次ソースで**誤検出と確認して却下**したもの。再発見時の再調査コストを防ぐため記録する。

| 指摘 | 却下理由 |
|---|---|
| 「`PRAGMA user_version={SCHEMA_VERSION}` はSQLi」 | `SCHEMA_VERSION` はモジュール定数の整数。そもそもPRAGMAはプレースホルダを使えない。攻撃経路なし |
| 「ログインにセッション固定の脆弱性」 | Flaskの署名Cookie方式にサーバ側セッションIDは存在せず、`session.clear()`→再設定が標準の対策。実装は正しい（[Flask docs](https://flask.palletsprojects.com/en/stable/web-security/)） |
| 「パスワード定期変更（90日）が無い」 | NIST SP 800-63B は**定期変更を推奨しない**（漏えい時のみ変更）。現行設計が正 |
| 「`normalize_remark` の正規表現が無効（引数誤り）」 | 実コード `re.sub(r"\n{3,}", "\n\n", text)` は正しい。調査エージェントの読み違い |
| 「備考表示にXSS。`\|safe\|escape` を追加せよ」 | Jinja2の自動エスケープで安全。提案の `\|safe` はむしろ**脆弱化**させる誤り |
| 「管理者初期PWが本番ログに出力される」 | 出力は開発モードで自動生成した時のみ。本番は `ADMIN_INIT_PASSWORD` 未設定なら起動拒否で、印字経路に到達しない |
| 「Werkzeugのハッシュ既定は危険」 | 既定は scrypt（十分強い）。明示指定は任意の防衛的改善に留まる |
| 「編集モードでユーザーを黙って上書きできる（データ喪失バグ）」 | 編集中はID欄がreadOnly＋サーバは `mode=edit` 時にフォームのusernameを無視し `original_username` を権威とする。上書き経路なし（視認性のみB-4で改善） |

## 4. 🔍 公開情報による事実確認（2026-07-02時点）

### CF-Connecting-IP の信頼（`TRUST_CF_CONNECTING_IP=1`）→ **現状維持が妥当**
- RenderはCloudflareを前段に持ち、実クライアントIPの取得に CF系ヘッダ（True-Client-IP / CF-Connecting-IP）を使うのが実運用上の構成。全インバウンドがエッジ経由である限り、ヘッダはエッジで上書きされ偽装できない。
- render.yaml の前提コメントと「デプロイ後に `/logs` のIPを確認して0/1を確定する」手順は既に完備。**コード変更不要**。Render外へ移設する場合に `0` へ戻す運用も文書化済み。
- 出典: [Cloudflare HTTP headers](https://developers.cloudflare.com/fundamentals/reference/http-headers/) / [Restoring original visitor IPs](https://developers.cloudflare.com/support/troubleshooting/restoring-visitor-ips/restoring-original-visitor-ips/) / [Render feedback: X-Forwarded-For](https://feedback.render.com/features/p/send-the-correct-xforwardedfor)

### 依存パッケージ → **全ピンにCVEなし・現状維持が妥当**
- Flask **3.1.3 = 最新**（CVE-2026-27205 対応済み）。Werkzeug `>=3.1.8` は safe_join 系CVE（3.1.5で修正）を包含。Jinja2 `>=3.1.6` はサンドボックス回避CVE対応済み。
- Flask-WTF 最新は 1.3.0、Flask-Limiter 最新は 4.1.1 だが、**現行ピン（1.2.1 / 3.8.0）に既知CVEはない**。メジャー更新の破壊リスクを避け、安定優先で維持（月次 `pip-audit` 運用でカバー）。
- 出典: [Flask releases](https://github.com/pallets/flask/releases) / [CVE-2026-27205](https://www.sentinelone.com/vulnerability-database/cve-2026-27205/) / [Snyk: flask](https://security.snyk.io/package/pip/flask) / [Snyk: flask-limiter](https://security.snyk.io/package/pip/flask-limiter) / [Snyk: Flask-WTF](https://security.snyk.io/package/pip/Flask-WTF)

## 5. ⚪ 精読して問題なしと確認した項目

- **認可:** 全19ルートのガード網羅（管理ルートは `deny_if_not_admin` に統一、IDORなし、`/staff/<u>` はDB照合）
- **注入・XSS:** SQL全文パラメータ化、`|safe`・`innerHTML` 不使用（JSは `textContent` のみ）、備考はNFC正規化＋制御文字除去＋500字上限
- **セッション:** `__Host-` Cookie（Secure/HttpOnly/SameSite=Lax）、無操作30分＋絶対24時間、ログイン時 `clear→再設定`、停止・削除の即時反映（毎リクエストDB再取得）
- **CSRF/CSP:** 全POSTにトークン、CSPは nonce方式で `unsafe-inline` なし・`frame-ancestors 'none'`・`form-action 'self'`
- **ブルートフォース:** IP単位レート制限＋ダミーハッシュ検証による応答時間平準化（ID列挙対策）＋弱PW拒否
- **データ保全:** WAL、`with conn` トランザクション、DELETE→INSERT一括の提出処理（重複行の発生経路なし）、整合性検査付きバックアップ（移行前/日次/月次/手動）＋24hスロットル＋ロック
- **UI/デザイン:** 色語彙の一貫性（緑=正常・橙=注意・赤=失敗・灰=中立）、ボタン階層、主要文字コントラストAA以上、44pxタップ領域、フォーカス可視、モーダルのフォーカストラップ、モバイル1カラム化
- **テスト:** 全ルートをカバー（実装後151 passed）。skip/xfailなし
