## GPT/Gemini レビュー統合版の判断（2026-06-24）

今回の統合判断では、Gemini案の「初心者向けに画面・文言・保守導線をさらに軽くする」方向性は採用する。一方で、現行コードと一次資料に照らすと、無条件に採用しない方がよい提案もある。

## Claude Code へ渡す実装前提

- 本アプリは Render Starter で公開中。Render公開を前提に、`TRUSTED_PROXY_HOPS=1` と永続ディスク `/var/data` の運用を維持する。
- パスワードは8文字以上の運用を維持する。NIST SP 800-63B の単独パスワード15文字推奨との差は、試用・小規模運用上の判断として受容し、8文字運用のまま補助的な安全策を強化する。
- 今回の実装候補は、画面・文言・保守導線・Host/プロキシ設定・バックアップ確認性の改善を主対象とする。大規模な設計変更、ドキュメント移動、依存ファイル統合は行わない。
- `CODE_REVIEW.md` は引き続き受け入れ基準として扱い、実装後は `README.md` / `IMPROVEMENTS.md` / アプリ内ヘルプを実装状態と同期する。

### 採用する考え方

- 画面は「次に押すボタン」が一目で分かる状態を維持する
- 説明文は短くし、不安をあおる言葉を避ける
- 管理者が月1回やることを、READMEだけでなく画面上でも確認しやすくする
- 備考など自由入力は、保存前・保存後・ログ表示の全段階で安全に扱う

### 修正して採用する考え方

- 色彩設計は「70:25:5」を目安にしてよいが、現在の緑は白文字とのコントラストが十分に高い。大きな色替えより、主要ボタン・警告・補助操作の役割を一貫させる。
- オレンジ系アクセントは全ての保存ボタンへ広げず、締め切り前の注意、バックアップ要確認、未提出など「注意を向けたい状態」に限定する。
- 文言削減は進めるが、シフト入力画面の「〇 出勤可」「× 休み」は業務判断そのものなので、意味が曖昧になるほど削らない。
- パスワード8文字運用は維持する。ただし `COMMON_PASSWORDS` は小規模なローカル拒否リストとして残し、最小リスト＋反復判定＋ID一致拒否＋管理画面での注意文を併用する。

### 現時点では採用しない考え方

- `CODE_REVIEW.md` と `IMPROVEMENTS.md` をすぐ `docs/` に移動しない。このリポジトリでは `CODE_REVIEW.md` が試用可否の受け入れ基準として機能しており、ルート直下に置く方が見落としにくい。
- `requirements.txt` と `requirements-dev.txt` は統合しない。本番依存と開発・監査依存を分けておく方が Render 本番環境を軽く保てる。
- `safe_ym` と `resolve_ym` は拙速に統合しない。GET表示用とPOST保存先確定用で役割が異なるため、統合するなら名前と責務を明確にした上でテストを先に置く。

## 次の改善プラン（統合優先度順）

### P1: Render公開前提の認証・信頼境界を固める

目的: Renderで公開中の小規模アプリとして、現在の運用方針を保ちながら認証・Host・プロキシ信頼境界を明確にする。

- パスワード8文字運用を維持したまま補助策を強化する
  - 現状: 8〜128文字、頻出値・ID一致・単純反復を拒否
  - 方針: 最小長は変更しない。代わりに、弱い値の拒否、説明文、管理者の再発行手順、監査ログを強化する
  - 実装案: パスワード入力欄の説明を「8文字以上。名前・ID・単純な数字は避けてください。」に統一する
  - 実装案: `COMMON_PASSWORDS` は残し、`shiftflow2026`、`password2026`、`admin1234` など実運用で選ばれそうな弱い候補を最小限追加する
  - 実装案: 管理者によるパスワード再設定時に、本人へ別経路で伝える注意を画面に短く表示する
  - 実装しないこと: 15文字強制、文字種強制、定期変更強制、外部漏えいDBへのパスワード送信
  - 追加テスト: 8文字の十分なパスワードは通る、頻出値・ID一致・単純反復・追加NG値は拒否される
- Render公開前提でプロキシ/IPヘッダの扱いを明確化する
  - 現状: `render.yaml` は `TRUSTED_PROXY_HOPS=1` と `TRUST_CF_CONNECTING_IP=1` を設定している
  - 方針: Render公開を前提にこの設定は維持する。ただし、ヘッダ信頼の根拠をREADMEとテストに明記する
  - 実装案: 管理メニューまたはREADMEに「ログのIPはRender/Cloudflare経由のヘッダを元に表示する」旨を追記する
  - 実装案: `/logs` のIPが不自然な場合に確認する手順をREADMEへ追加する
  - 実装案: `TRUST_CF_CONNECTING_IP=1` の時も不正なIP形式は採用しない現行挙動をテストで維持する
  - 追加テスト: 信頼ONでは妥当な `CF-Connecting-IP` を採用、信頼OFFでは無視、不正値は `remote_addr` へフォールバック
- Hostヘッダ検証を追加する
  - 現状: Flaskの `TRUSTED_HOSTS` 相当が未設定
  - 方針: Renderの正規URLと、利用している場合はカスタムドメインだけを許可する
  - 実装案: `TRUSTED_HOSTS` 環境変数を追加し、本番では未設定なら起動停止
  - 例: `TRUSTED_HOSTS=shift-flow.onrender.com,example.com`
  - 実装案: `render.yaml` に `TRUSTED_HOSTS` の設定枠を追加する。実際のRenderサービス名に合わせて値を確認してから設定する
  - 追加テスト: 許可Hostは200、不許可Hostは400系、開発環境はlocalhost/127.0.0.1を許可

### P1: バックアップ運用を初心者が画面だけで判断できるようにする

目的: 「バックアップはあるはず」ではなく「正常・要対応・次にすること」が見える状態にする。

- 管理メニューへバックアップ状態パネルを追加する
  - 表示する内容: 最終成功日時、最新ファイル名、最終エラー、ディスク空き、月次保存の確認日
  - 表示文例: 「バックアップ正常」「月次保存がまだです」「復元練習を確認してください」
- 月次チェックをDBに記録する
  - `app_state` に `last_external_backup_checked` と `last_restore_drill_checked` を保存
  - 管理者がチェックボタンを押すだけで月次確認を残せるようにする
- READMEの復元手順を画面と一致させる
  - 本番DBを直接触る手順と、複製DBで練習する手順を分ける
  - WAL/SHMを新DBへ流用しない注意を画面にも短く出す

### P2: UI/UXを「迷わない・押しやすい・戻れる」に寄せる

目的: 見た目の刷新ではなく、現場で迷う時間と誤操作を減らす。

- 44px以上の操作領域を全リンクへ広げる
  - 現状: 主要ボタンは概ね十分。メニュー下部のテキストリンクは小さめ
  - 実装案: `footer-nav` のリンクと `link-button` に `min-height: 44px` と十分な余白を付ける
- メニュー下部のリンクとログアウトを同じ行・同じ高さへ揃える
  - 現状: 本番Safariで「パスワード変更」「使い方」と「ログアウト」の行・下線・高さがずれて見える
  - 原因候補: `a` と `form > button` の表示方式、余白、行高、下線の当たり方が揃っていない
  - 実装案: `footer-nav` 内の `a`、`form`、`.link-button` を同じ `inline-flex`/`align-items: center`/`min-height`/`line-height` に統一する
  - 実装案: `form` の余白を0にし、下線は要素ごとの差が出にくい `text-decoration` か `border-bottom` に統一する
- ボタン階層を固定する
  - Primary: 保存、ログイン、提出状況へ進む
  - Secondary: 戻る、一覧を見る、使い方
  - Danger: 停止、解除、失敗対応
  - Link: 低頻度の補助操作だけ
- 色の役割を固定する
  - 緑: 通常の主要操作、正常
  - 黄/オレンジ: 未提出、締め切り前、バックアップ要確認
  - 赤: 失敗、権限エラー、停止
  - 青: フォーカス表示のみ
- シフト入力の状態をより分かりやすくする
  - 現状の「〇 出勤可」「× 休み」は維持
  - 保存前に「今月の入力日数」「備考ありの日数」を短く表示する
  - 締め切り後は「見るだけ」ラベルを画面上部と保存ボタン位置の両方で示す
- 「みんなの希望」カレンダーの職員表示を枠内に収める
  - 現状: 本番Safariで職員名と表示色がカレンダーのセルからずれたり、名前が不自然に折り返されて見える
  - 原因候補: `.shift-tag` の幅、色見本、名前、管理者印の並びがセル幅に対して固定されておらず、長い日本語名や複数人表示で崩れる
  - 実装案: `.calendar-table` は列幅とセル内余白を安定させ、セル内の職員リストは縦積みのコンパクト表示にする
  - 実装案: `.shift-tag` は「固定幅の色見本」「名前」「補助印」を1つの部品として扱い、`min-width: 0`、`overflow-wrap: anywhere`、必要なら省略表示でセル外にはみ出させない
  - 実装案: 390px前後では表全体を無理に縮めず、横スクロールまたは日別リスト表示へ切り替える。色見本と名前が重ならないことを最優先にする
  - 追加確認: 2名以上が同じ日に入る、管理者印が付く、長い職員名を使う、390px幅とデスクトップ幅で崩れない
- ユーザー管理の保護中管理者行を他の職員行と揃える
  - 現状: 保護される管理者の「修正」ボタンと「保護」表示の位置が、他の「修正」「停止」行とずれて見える
  - 原因候補: `row-actions` 内で、ボタンと保護ラベルが別サイズ・別配置になっている
  - 実装案: 保護ラベルを「停止ボタンと同じ枠を占めるステータス表示」にし、`修正` と `保護` が2列で揃うようにする
  - 実装案: デスクトップでは右端2列、モバイルでは同じ幅の2ボタン風レイアウトにし、`aria-disabled` または説明テキストで「停止できない管理者」であることを伝える
- 管理者の月次導線を1本にする
  - メニュー上では「提出状況・締め切り」を最優先
  - `みんなの希望を見る` は提出締め切り後の確認として位置づける

### P2: 文言を短くしつつ、原因と次の操作は残す

  - 現状例: 「この画面は開けません」
  - 改善案: 「管理者だけが使う画面です。」
- 締め切り系
  - 現状例: 「締め切り済みのため変更できません。」
  - 改善案: 「締め切り後のため、見るだけです。変更は管理者に伝えてください。」
- ユーザー管理系
  - 現状例: 「対象の職員が見つかりません（停止中の可能性があります）。」
  - 改善案: 「職員が見つかりません。停止中の職員は一覧で確認してください。」
- バックアップ系
  - 改善案: 「バックアップを確認してください。最新の保存に失敗しています。」

### P2: 自由入力とログの安全性をもう一段固める

目的: 備考に個人情報を書かない前提を支援し、表示崩れやログ汚染を防ぐ。

- 備考の保存前正規化を追加する
  - Unicode NFC正規化
  - 改行は最大数を制限
  - 制御文字は削除または空白化
  - 500文字上限は維持
- 備考の画面表示をさらに守る
  - 現状: Jinjaの自動エスケープでHTML注入は抑止
  - 追加案: 長い連続文字の折り返しを全表示箇所で統一
- 監査ログのdetail生成を型で分ける
  - パスワード、備考本文、CSRFトークンは記録しない方針をテストで固定する
  - detailのJSON化処理を小関数化し、長さ上限とキー制限を一箇所に寄せる

### P3: 保守しやすい構成へ少しずつ整える

目的: 初心者管理者が触る文書は少なく、開発者が読むコードは追いやすくする。

- `app.py` の分割は段階的に行う
  - すぐ大規模なBlueprint分割はしない
  - 先に `auth`, `users`, `shifts`, `operations` の関数群へコメント見出しとテスト対応を整える
  - ルート数や変更頻度が増えた時点でBlueprint化する
- 年月処理を整理する
  - `safe_ym`: 表示用GETパラメータ
  - `resolve_ym`: POST保存先をhidden値込みで確定
  - 改善案: 統合ではなく、共通の低レベル関数 `coerce_year_month` を作り、用途別関数は残す
- テストコメントを現行仕様へ更新する
  - `must_change_password` 互換列の説明
  - 初回強制変更廃止後のログインフロー
  - 旧 `confirmed_shifts` 温存方針
- 旧URL互換の削除期限を決める
  - `/confirm` と `/confirmed` は次回または次々回リリースで削除候補
  - 削除前にREADMEとログでアクセス有無を確認する
